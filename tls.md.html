## Basic TLS在TCP协议层之上。 TLS是SSL的IETF标准化，SSL原用于web，两者有区别不能互操作，TLS的主要目标是使SSL更安全，并使协议的规范更精确和完善。 另有无线环境下的WTLS，以及UDP上的DTLS。 ``` +-----------------------------------------------------------------+ |--------------------------------------------+ | | Handshake | Change Cipher Spec | Alert | application data | | Protocol | Protocol | Protocol | protocol(HTTP) | |--------------------------------------------------+ | | Record Protocol | | |-----------------------------------------------------------------| | TCP | +-----------------------------------------------------------------+ ``` ## 记录协议 Record Protocol 记录协议是一种分层协议，每一层中的信息可能包含长度、描述和内容等字段。 记录协议支持信息传输，从应用层接收任意大小无空块的连续数据、将数据分段到可处理块、压缩数据、应用 MAC、加密以及传输结果等。对接收到的数据进行解密、校验、解压缩、重组等，然后将它们传送到应用层。 提供的连接安全性具有两个基本特性： 私有 ― 对称加密用以数据加密（DES、RC4 等）。对称加密所产生的密钥对每个连接都是唯一的，且此密钥基于另一个协议（如握手协议）协商。记录协议也可以不加密使用。 可靠 ― 信息传输包括使用密钥的 MAC 进行信息完整性检查。安全哈希功能（SHA、MD5 等）用于 MAC 计算。记录协议在没有 MAC 的情况下也能操作，但一般只能用于这种模式，即有另一个协议正在使用记录协议传输协商安全参数。 安全参数(security parameters): connection end: server, client bulk encryption algorithm: null, rc4, rc2, des, 3des, des40, idea, aes MAC algorithm: null, md5, sha compression algorithm: null(0), (255) master secret client random server random 通过安全参数生成的密钥： client write MAC secret server write MAC secret client write key server write key 连接状态(connection state): compression state cipher state MAC secret sequence number ## 握手协议 Handshake Protocol TLS 握手协议由三个子协议组构成(Handshake, Change Cipher Spec, Alert)，允许对等双方在记录层的安全参数上达成一致、彼此之间相互认证、协商安全参数、互相报告出错条件。 TLS 握手协议提供的连接安全具有三个基本属性： 可以使用非对称的，或公共密钥的密码术来认证对等方的身份。该认证是可选的，但至少需要一个结点方。 共享加密密钥的协商是安全的。对偷窃者来说协商加密是难以获得的。此外经过认证过的连接不能获得加密，即使是进入连接中间的攻击者也不能。 协商是可靠的。没有经过通信方成员的检测，任何攻击者都不能修改通信协商。 协商会话的具体内容： session identifier peer certificate compression method cipher spec master secret is resumable 它们被用来生成安全参数为记录协议层所用。 ### 协商消息 ``` Client Server ClientHello --------> ServerHello Certificate* ServerKeyExchange* CertificateRequest* <-------- ServerHelloDone Certificate* ClientKeyExchange CertificateVerify* [ChangeCipherSpec] Finished --------> [ChangeCipherSpec] <-------- Finished Application Data <-------> Application Data *为可选消息 ``` 缺省只进行Server端的认证，Client的认证是可选: 1. Client在TCP链接建立之后，发出一个ClientHello来发起握手，这个消息里面包含了自己可实现的算法列表和其它一些需要的消息， 1. Server会回应一个ServerHello，这里面确定了这次通信所需要的算法，然后发过去自己的证书（里面包含了身份和自己的公钥）, 1. Client在收到这个消息并验证真实性，然后会生成一个秘密消息ClientKeyExchange，包含per_master_secret用于生成加密验证的密钥，用Server的公钥加密后传过去， 1. Server用自己的私钥解密后，会话密钥协商成功，双方可以用同一份会话密钥来通信了。 ## Reference http://tools.ietf.org/html/rfc4346
