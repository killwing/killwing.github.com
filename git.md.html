## 环境与配置 版本库目录`.git`在工作区根目录下，在子目录下操作时Git会递归向上查找`.git`目录。 `git init`在当前目录下创建并初始化新的版本库。 ### 配置文件`.gitconfig` 配置文件有三个级别，优先级从高到低依次为版本库`.git/config`，全局`~/.gitconfig`，系统`/etc/gitconfig`。 配置文件格式为INI格式，使用`git config`操作，也可以操作其它任何INI格式的文件。 配置文件例子： ``` [user] name = kelvin # 提交中作者和提交者的ID email = kelvin@company.com # 提交中作者和提交者的电邮 [merge] tool = extMerge # 外部合并工具 [mergetool "extMerge"] cmd = extMerge "$BASE" "$LOCAL" "$REMOTE" "$MERGED" trustExitCode = false [diff] external = extDiff # 外部比较工具 [color] ui = true # 在终端显示颜色 [alias] # 别名 lg = log --graph --oneline --all --decorate st = status co = checkout ci = commit br = branch sm = submodule [core] autocrlf = false # 不修改换行符 safecrlf = true # 如果文件中有不同换行符则拒绝提交 whitespace = blank-at-eol,tab-in-indent,tabwidth=4 # 检测空白的规则：行尾有空白，缩进有tab，并设tab为4个字符 ``` ### 忽略文件`.gitignore` 作用范围是所在目录及其子目录，只对未跟踪的文件有效，对已经加入版本库的文件无效。 所有空行或者以注释符号`#`开头的行都会被忽略。 可以使用标准的[glob](http://man7.org/linux/man-pages/man7/glob.7.html)模式匹配。 匹配模式最后跟反斜杠`/`，说明要忽略的是目录。 如果名称前面是个`/`，说明要忽略的文件在此目录下，而非子目录的文件。 要忽略指定模式以外的文件或目录，可以在模式前加上`!`取反。 独享式忽略可以使用`.git/info/exclude`文件设置，或者配置变量`core.excludesfile`。 ### 属性文件`.gitattributes` 为版本库中的文件或目录添加属性，格式：`<pattern> <attr1> <attr2>` * `pattern`类似忽略文件。 * 一般`attr`表示设置属性为true，`-attr`设为false取反，`!attr`不设置，而是根据git配置里的值，`attr=`设置具体枚举值。 * 设置宏，如`[attr]binary -text -diff`。（仅在工作区根目录的属性文件） 属性文件的级别，优先级从高到低依次为版本库`.git/info/attributes`，工作区同一目录下`.gitattributes`的并递归向上查找至工作区根目录，全局由`core.attributesfile`配置指定（如`~/.gitattributes`），系统`/etc/gitattributes`。 `git check-attr` 查看属性设置 常用属性： * `text` 显式指定文件的类型，`text`文本文件，`-text`二进制，`text=auto`智能判别。 * `eol` 设定文本文件的换行符格式`crlf`或`lf`，如果没有`text`属性时会设置`text`属性。 * `ident` 开启文本文件中的关键字扩展，即用blob的SHA-1值替换文本中的`$id$`字段。 * `filter` 自定义转换过滤器，以便文件在检入到版本库以及检出到工作区时进行相应的转换，通过git配置文件来完成。 * `diff` 定义比较方式，`diff`文本方式，`-diff`二进制方式，`!diff`智能判别，`diff=<driver>`设定一个外部驱动如`word`根据git配置来比较。 * `merge` 指定合并策略，`merge`使用内置的三向合并策略，`-merge`二进制文件合并策略，`!merge`和`merge`类似但git配置了`merge.default`就用配置的策略，`merge=<driver>`使用指定的合并驱动执行三向合并。 * `whitespace` 对文本中的空白符进行检查。`whitespace`开启所有的空白符检查，`-whitespace`不检查，`!whitespace`使用`core.whitespace`配置检查，`whitespace=`设置检查项。 * `export-igonre` 在执行`git archive`时不导出。 * `export-subst` 在执行`git archive`时展开相应文件内容的占位符`$Format:Placeholders$`。 * `delta` `-delta`则不对该路径指向的blob文件执行delta压缩。 * `encoding` 设置文件所用的字符集。 * `binary` 宏，相当于`-text`和`-diff`。 ### 协议与代理 git传输可以用`ssh`, `http(s)`, `git`（只读）, `file:///`, `/path`智能协议（`http`需配置），以及`ftp`, `rsync`哑协议。哑协议的缺点是传输速度慢效率低。 对于`ssh`/`http`协议代理，只要设置`http_proxy`环境变量即可。 对于`git`协议代理，[设置core.gitproxy配置命令调用socat或corkscrew](http://www.emilsit.net/blog/archives/how-to-use-the-git-protocol-through-a-http-connect-proxy/)。 ## 暂存区 暂存区（stage, 又称索引index）：一个包含文件索引的目录树`.git/index`，作为下次提交的内容，提交之后stage内容就和HAED一致。 `git ls-files -s`查看stage的目录树，用`git write-tree`把当前stage内容生成一个树对象，然后可以用`git ls-tree`查看树对象。 `git status`查看当前状态。 状态转换： [workspace] -- `add` --> [stage] -- `commit` --> [repo] [workspace] <-- `checkout --` -- [stage] <-- `reset --` -- [repo] `git diff`比较([unified格式](http://stackoverflow.com/questions/2529441/how-to-work-with-diff-representation-in-git))。`--check`会把可能的多余白字符修正列出来。`--word-diff`逐词比较。`--binary`二进制文件比较。 比较各种状态： [workspace] <-- `diff` --> [stage] <-- `diff --cached` --> [HEAD] [workspace] <-- `diff <commit>` --> [commit] [stage] <-- `diff --cached <commit>` --> [commit] 包含路径`git diff <commit1> <commit2> -- path`比较文件版本差异。 可在版本库之外使用`git diff <path1> <path2>`来比较其它文件。 ## 增删改 `git add` 将文件加入版本库。`-u`选项可以将被版本库追踪的本地文件的修改和删除全部记录到stage中。`-A`记录所有工作区的改动。 `git rm` 从版本库中删除文件，相当于本地删除后`add -u`。`--cached`也从stage移除文件(unstage & untrack)。 `git mv` 重命名，相当于本地改名后`add -A`（先删除，再加入），会自动比较文件相似度以判断是不是重命名。 `git commit` 把stage中的改动提交到版本库。`-a`可以自动把修改和删除的文件加入索引并提交，新加的文件不受影响。`--amend`修改最后一次提交，即撤销后重新提交。`-C`重用某个提交的作者和注释。 `git ls-files` 查看加入到版本库中的所有文件，默认HEAD。 ## 对象 四种对象，位置在`.git/objects/`，ID前2位作为目录名，后38位作为文件名。 * 提交对象(commit): 包括一个树对象，一个或多个父提交对象，作者和提交者，以及注释。 * 树对象(tree): 包括文件（类型：blob），目录（类型：tree），它们的ID，以及读写权限。 * 二进制对象(blob): 具体文件内容。 * 标签对象(tag): 轻量级标签就是一个提交对象。 `git cat-file` 查看对象的类型和内容。 `git hash-object` 直接从文件内容创建对象。 `git show-ref` 查看所包含的引用。 对象ID: 40位SHA1 hash值。不必写全，至少4个字符，没有冲突即可。 ID的计算：`sha1(对象类型<space>对象长度<null>对象内容)`，例如用命令生成：`(printf "commit 123\0"; git cat-file commit HEAD)|sha1sum` ### 对象指代（以HAED为例） * `HAED`: 位置在`.git/HEAD`，默认指向`.git/refs/heads/master`(当前分支，这里是master)，类似链表的头指针。当执行提交时，HEAD指向的提交将作为新提交的父提交。 * `HAED^`: 父提交，`HEAD^^^`或`HEAD~3`都指第前3次父提交。也可用类似reflog表示法：`HEAD@{3}`。 * `HAED^{tree}`: 提交对应的树对象。 * `HEAD:path/to/file`: 提交对应的文件对象。 * `:path/to/file`: stage中的文件对象。 * `:/"commit xxx"`: 日志中查找字符串对应的提交。 * 如果A是含附注说明的tag，则`A^{}`, `A^{commit}`, `A^0`, `A~0`指它对应的提交。其他类似：`A^{tree}`, `A:path/to/file`。 ### 提交范围 * `A`: 单独表示该版本开始的所有能追溯到的历史提交 * `A^@`: 自身除外 * `A^!`: 表示自身 * `A B`: 相当于单独使用时的并集 * `A..B`: 从A到B的范围，指从B可以追溯到而不能从A追溯到，不包括A本身(A, B]，B可省默认为HEAD。相当于`^A B`或`B ^A`或`B --not A`，`^`表示从其中排除。 * `A...B`: 排除A和B能共同追溯到的版本，顺序无关 `git rev-parse`查看具体提交的ID（也可查看引用和对象的ID），`git rev-list`查看提交范围，`git log`接提交范围（默认HAED）以显示提交历史。 ### 对象管理 `git fsck` 查看版本库中包含的没有被任何引用关联的对象。还被reflog引用到的不会显示，可以用`git reflog expire=<date> --all`强制让reflog在date之前的记录全部过期。 `git prune` 清除没有被任何引用关联的对象。 `git pack-refs` 对引用文件进行打包到`.git/packed-refs`文件中。 `git repack` 打包所有有引用关联的对象到`.git/objects/pack/pack-*.pack`中，index文件为`.git/objects/pack/pack-*.idx`。 `git gc` 优化存储包括打包引用和对象，丢弃90天前的reflog记录，清除2周前未被关联的对象，以及过期合并冲突历史记录等其它清理。有些情况下如接收推送后会自动执行`git gc --auto`进行清理。 ## 重置与检出 `git reset`重置当前分支引用（最新提交，HAED也随之改变）。 两种用法： * 包含路径`<commit> -- <path>` 不会重置引用，而是用此commit（省略则默认为HEAD）下的文件path替换掉stage中的文件。 * 不包含路径： * `--hard <commit>` 重置引用，并替换stage和workspace与此commit一致 * `--soft <commit>` 重置引用，不改变stage和workspace * `--mixed <commit>` 默认模式，重置引用，并替换stage与此commit一致，但不改变workspace 如果发生了引用重置，提交历史也会被重置，可用`git reflog`查找以前的提交ID恢复，用`<refname>@{<n>}`表示refname之前n次改变时的ID。 `git checkout`重置HEAD的引用（当前分支改变，但HEAD总是指向当前分支的头）。 两种用法： * 包含路径`<commit> -- <path>` 不会改变HEAD头指针，而是用此commit下的文件path替换掉stage和workspace中的文件，commit若省略则相当于从stage中替换掉workspace中的文件。 * 不包含路径： * `<commit>` 重置HEAD为具体的commit（也可以是引用比如某个tag），不再指向当前分支引用(detached HEAD)。 * `<branch>` 切换到某个分支（加`-b`选项可以新建并切换分支），对提交进行跟踪（不处于detached HEAD状态），并用此branch的树更新stage和workspace中的文件。 * 接无参数或HEAD可以汇总显示workspace, stage和HEAD的差异。 detached HAED状态下的提交不被任何分支跟踪到，当reflog中含有此提交的日志过期后（默认90天），此提交随时都会被从版本库中删除。可以通过合并`git merge`来保存这些提交到具体的分支上。 重置一般用来覆盖暂存区，而检出一般用来覆盖工作区。 ## 存档 `git stash` 保存当前工作状态为一个存档，分别对workspace和stage进行保存，然后把当前状态恢复为修改前的样子（使用了`git reset --hard HEAD`）。`apply/pop --index`应用存档被暂存的更改。`branch`创建一个新的分支来重新应用存档的更改。 还没有加入版本控制的文件不能被保存。 每个存档的标识为`stash@{<n>}`格式，stash是用引用`.git/refs/stash`和引用变更日志`.git/logs/refs/stash`实现的，可以用`git reflog`查看。 ## Debugging `git blame` 查看文件每一行最后修改的版本和作者，`^`开头的那些行指初次提交的内容且没被修改过。`-C`分析标注的文件然后尝试找出其中代码片段的原始出处/提交。 `git bisect` 自动对提交二分查找，通过不断测试不同提交的代码找到引入bug的提交。 步骤： 1. `bisect start` 开始查找 1. `bisect good` 标记好提交 1. `bisect bad` 标记坏提交 1. 不断重复good/bad进行二分查找，最终定位到引入bug的坏提交 1. `checkout bisect/bad` 切换到最终定位的坏提交，由引用`.git/refs/bisect/bad`标识 1. 定位和修复bug 1. `bisect reset` 撤销查找时的临时文件和引用，切换回原来分支 `bisect run`用脚本自动执行来代替步骤2-4来标记提交： * 退出码0：good * 退出码1~127(125除外)：bad * 退出码125：skip ## 标签 又称里程碑，位置在`.git/refs/tags/`，记录了所有标签。 `git tag` 显示标签，创建标签，删除标签（`-d`）。 三种类型： * 轻量级标签（不带选项创建）：就是一个提交（对象）。 * 带附注说明的标签（`-a/-m`）：标签对象，含有名字，对应的提交，创建用户(tagger)，时间，以及说明。 * 带签名的标签（`-s/-u`）：带附注说明的标签并包含GnuPG签名。`-v`验证签名的有效性。 标签共享： * 推送：必须显式指定或用`--tags`选项。 * 获取或拉回：自动同步远程标签（仅获取分支），如果已有同名的则不会同步。 `git describe` 将提交显示为一个易记的名称，如果恰好有tag则为tag的名字，否则格式为：`基础版本号(最近的tag)-距离(距此tag的提交次数)-g提交ID`，可以作为软件的版本号。`--tags`对轻量级标签也有效，默认不会使用。 `git name-rev` 显示提交ID及其对应的一个引用，默认优先使用分支名。`--tags`优先使用标签。 ## 分支 分支: 位置在`.git/refs/heads/`目录下的引用(ref)，内容是分支的最新提交。 引用日志(reflog): 位置在`.git/logs/refs/heads/`，记录了分支的所有提交变更，默认非裸(bare)版本库都提供。 `git branch` 显示分支（`*`表示当前分支），创建分支，删除分支（`-d/-D`），改名分支（`-m/-M`）。`-b`创建并切换分支。`-v`查看各个分支最后一次提交信息。`--merged`查看哪些分支已被并入当前分支。`--no-merged`查看哪些分支还没被并入当前分支。 ## 衍合 `git rebase` 将指定范围的提交重新一个个应用到另外一个提交之上。 用法：`git rebase [--onto <newbase>] <upstream> [<branch>]` 相当于过程： 1. `git checkout <branch>`，branch默认为HEAD，不提供则留在当前分支上。 1. 把`<upstream>..<branch>`的变化保存到一个临时区域。 1. `git reset --HARD <newbase>`，如果没有`--onto`指明newbase，则用upstream。 1. 将临时区域的提交列表按顺序逐一重新应用到重置后的分支上，如果提交已在分支上包含则跳过此提交。 1. 如果遇到冲突则暂停，用户将冲突解决后可以`rebase --continue`继续，`rebase --skip`跳过，`rebase --abort`终止并切换到rebase前的分支上。 用`-i`选项使用交互式rebase（根据文件中的动作顺序执行），可以编辑文件改变提交顺序以及删除行以跳过提交，动作： * pick: 应用此提交 * reword: 应用提交并修改提交说明 * edit: 应用提交后暂停，以便使用`git commit --amend`对提交进行修补 * squash: 该提交会与前面的提交压缩为一个 * fixup: 同squash，但此提交的说明被丢弃 * x cmd: 执行一个shell命令，出错的话会暂停 永远不要衍合那些已经推送到公共仓库的更新。衍合是用新提交代替旧提交，如果已经有人基于旧提交工作，则必须再重新合并同样内容的新提交。 ## 合并 `git merge <commit>` 将commit对应目录树和当前分支的目录树的内容进行合并，合并后新的提交以当前分支的提交为第一个父提交，被合并的为第二个父提交。 快进式(fast-forward)合并：当前分支的最新提交是要被合并分支最新提交的祖先提交。这时不会产生新的合并提交，而是直接将HEAD指向被合并分支的最新提交。可用`--no-ff`选项禁止。 合并策略： * resolve：只用于合并两个头，使用三向合并策略。最快最安全。 * recursive：只用于合并两个头（默认），使用三向合并策略。冲突最小化，可以发现由于重命名导致的冲突。 * ours：冲突时选择当前版本。 * theirs：冲突时选择被合并的版本。 * subtree[=path]：使用子树合并策略。 * octopus：合并两个以上的头（默认），拒绝需要手动解决的复制合并。 * ours：合并两个以上的头，总是取当前分支的内容。 * subtree：调整过的recursive策略。 ### 冲突 冲突信息： * `.git/MERGE_HEAD` 合并的提交ID * `.git/MERGE_MSG` 合并失败的信息 * `.git/MERGE_MODE` 合并状态 `git status`查看发生了哪些冲突。 `git ls-files -s`的第四个字段是stage编号，当发生冲突时会有0以上的编号，在冲突解决后消失： 1. `:1:<filename>` 冲突文件修改之前的版本，冲突双方的共同祖先版本(base) 1. `:2:<filename>` 冲突文件在当前分支中的修改版本(local) 1. `:3:<filename>` 冲突文件在被合并分支中的修改版本(remote) 工作区的版本冲突的地方用特殊标记标识，需要用手工编辑选取版本并加到stage以解决： ``` <<<<<<< HEAD 当前分支更改 ======= 被合并分支更改 >>>>>>> ID ``` `git mergetool` 用图形工具解决冲突。也可以用来交互式解决树冲突。 `git cherry-pick` 将一个提交导出为补丁文件，然后在当前HAED上重放。 `git revert` 用一个新提交来撤销一个提交的变化。 ## 远程 `git clone` 复制一个版本库，注册上游版本库为`origin`，并对master分支建立跟踪。`--bare`复制为一个没有工作区的裸版本库（一般目录以.git为后缀），不会建立远程分支的跟踪而只是复制到本地。`--mirror`复制一个裸版本库并对上游版本库进行了注册。 `git remote` 远程版本库管理。`add`新注册远程版本库，`rm`删除远程版本库，`rename`重命名远程版本库，`-v`显示所有已注册的版本库，`show <remote>`查看remote的详细信息。 `git ls-remote` 查看远程版本库的引用。 ### 远程分支 位置`.git/refs/remotes/origin/`，查看`git branch -r`。 对应`.git/config`中的配置： ``` [remote "origin"] fetch = +refs/heads/*:refs/remotes/origin/* url = http://path/to/repos/project.git ``` 远程分支不是真正的分支，只是类似标签的引用，如果需要修改得创建新的本地分支： `git checkout remote-branch-name`相当于`git checkout -b remote-branch-name origin/remote-branch-name`，创建了同名的本地分支并切换到上面，同时建立了本地分支和远程分支的跟踪。基于本地分支创建则不会自动建立跟踪，可以使用`--track`选项在创建分支时建立跟踪。 对应`.git/config`中的配置： ``` [branch "remote-branch-name"] remote = origin merge = refs/heads/remote-branch-name ``` 跟踪特性： * `git status`会显示本地分支和远程分支提交之间的关系。 * `git pull`会和被跟踪的远程分支进行合并或衍合。 * `git push`会推送到远程版本库的同名分支中。 ### 推送与拉回 `git push <remote>` 推送当前分支更新。只有当前分支建立了远程跟踪才会被推送，否则需要用`git push <remote> <new-branch>`显式在远程创建同名分支，加上`-u`自动建立跟踪。 推送目的位置： * 远程版本库：配置里的`branch.<branchname>.remote`版本库，没有则用`origin`。 * 远程地址：配置里的`remote.<remote>.pushurl`地址，没有则用`remote.<remote>.url`。 * 远程分支：配置里的`remote.<remote>.push`引用表达式，没有则用`:`表示同名分支推送，没有同名分支则不推送。 默认不能向一个非裸版本库推送更新。 快进式推送：远程版本库相应分支的最新提交是本地版本库最新提交的祖先提交，合并时采用快进式合并。 如果远程版本库在配置里禁止了非快进式推送`receive.denyNonFastForwards`，这时就需要先拉回远程的更新合并后再推送。 使用引用表达式`:refname`可以在推送时删除远程分支或标签。 `git pull` 获取更新，并合并当前分支更新。(`git fetch` + `git merge/rebase`) 拉回位置： * 远程版本库：配置里的`branch.<branchname>.remote`版本库，没有则用`origin`。 * 远程地址：配置里的`remote.<remote>.url`地址。 * 获取内容：配置里的`remote.<remote>.fetch`引用表达式。 * 需合并的本地分支：配置里的`branch.<branchname>.merge`。如果`branch.<branchname>.rebase`为true，则使用rebase代替合并，将本地分支的改动rebase到跟踪分支上。 `git fetch` 从远程版本库获取更新，默认从`origin`获取。`--no-tages`只获取分支和提交，不获取标签。`git remote upate`获取所有远程版本库的更新。 例如根据上面的配置`git fetch`就相当于`git fetch origin +refs/heads/*:refs/remotes/origin/*`，将远程版本库的所有分支复制为本地的远程分支，`+`表示强制引用替换即使是非快进式的。 ## 补丁 `git format-patch` 从提交范围创建对应的补丁，支持二进制文件。`-s`加上用户签名(signed-off)。 `git send-mail` 发送补丁。 `git am` 应用补丁。可以是mbox邮件补丁格式。 `git apply` 应用一般格式的补丁，但需手动提交，也不能保留补丁中的作者信息。类似Gnu patch。`-R`反向应用补丁。 ## 子模块 `git submodule` 将一个版本库作为另一个版本库的子目录，并保持各自提交相对独立不互相影响。`add`加入一个子模块（只是加了个链接）。`init`是将`.gitmodules`里的name和url信息注册到`.git/config`，如果已有则不会修改，一般clone一个带子模块的版本库后需要这样做。`update`根据`.git/config`clone新增的子模块或使现有子模块固定于某个提交(detached HAED)。`status`查看状态，前面有`-`说明还没有clone，`+`表示有新的提交。 `.gitmodules`配置文件，保存了子模块项目的URL和本地保存目录。 ``` [submodule "moduleA"] path = module/A url = /path/to/repos/moduleA.git ``` 关于子模块的重要一点（子模块的状态）：记录他们当前确切所处的提交（固定于某个提交，不随外部自动更新），并将这个提交记录为一个目录项而不是子目录或者文件(gitlink)。 更新子模块：要使所有子模块同步到远程最新，用`git submodule foreach git pull`。 删除子模块： * 删除`.gitmodules`里的相关项 * 删除`.git/config`里的相关项 * `git rm --cached path/to/submodule` (没有最后的slash) * 提交然后删除未加入版本库的子模块文件 ## 子树 `git read-tree` 将一个分支内容读到当前分支的一个子目录下（只更新在stage中）。 `git write-tree` 将stage中的树保存为对象。 `git commit-tree` 从目录树树对象创建提交，可以用来创建根提交。`-p`指定父提交。 `git ls-tree` 查看树对象。 子树合并：外部版本库作为一个目录被整个复制到本版本库中，并复制到本版本库中的子目录下的数据可以和原版本库建立跟踪关联。 步骤： 1. 创建本地分支和外部版本库跟踪。 1. `git read-tree`将此分支的树内容更新到当前分支上。 1. `git write-tree`将更新过后的当前分支目录树保存为树对象。 1. `git commit-tree`将保存的数对象提交并以这个本地分支和当前分支的最新提交作为父提交。 1. 重置当前分支到这个合并树提交。 合并后更新外部版本库上游改动：先更新对应的本地分支，再利用子树合并策略合并到当前分支上`git merge -Xsubtree=path branch`。 利用[git-subtree](https://github.com/apenwarr/git-subtree)插件方便地完成子树合并，拆分等操作。 ## 其它 ### 钩子 位置`.git/hooks/`，当执行特定的操作时会调用特定的钩子脚本，这些脚本在clone时不会被复制。 `git commit`调用 * `pre-commit` 在键入提交信息前运行，被用来检查即将提交的快照，当从该钩子返回非零值时，会放弃此次提交。 * `prepare-commit-msg` 在提交信息编辑器显示之前，默认信息被创建之后运行。 * `commit-msg` 接收一个参数，此参数是包含最近提交信息的临时文件的路径。如果该钩子脚本以非零退出，会放弃提交。 * `post-commit` 在整个提交过程完成后运行。 `git am`调用 * `applypatch-msg` * `pre-applypatch` * `post-applypatch` `git rebase`调用 * `pre-rebase` `git checkout`调用 * `post-checkout` `git merge`调用 * `post-merge` `git receive-pack`调用 * `pre-receive` * `post-receive` * `update` * `post-update` `git gc --auto`调用 * `pre-auto-gc` `git commit --amend` / `git rebase` 重写提交的命令调用 * `post-rewrite` ### 模板 在`git init`或`git clone`创建版本库时，会从模板目录复制钩子脚本`.git/hooks/*`，忽略文件`.git/info/exclude`及其他文件。 模板目录可以由选项`--template=<DIR>`，环境变量`$GIT_TEMPLATE_DIR`，配置变量`init.templatedir`指定，否则使用默认的安装目录下的模板目录。 ### 稀疏检出 默认在`git init`或`git clone`创建版本库时，如果不是一个裸库`--bare`且没有`-n`选项，会从版本库`.git/`检出所有文件到工作区。 如果只要检出部分文件，要设置`core.sparseCheckout`配置变量，并配置`.git/info/sparse-checkout`文件（类似忽略文件），然后执行`git checkout`。 `git ls-files -s -v`查看状态，`H`表示文件已被暂存，`S`表示被忽略检出的文件。 ### 浅克隆 `git clone/fetch --depth <depth>` 只获取版本库的一部分即最近的depth+1个历史提交。源版本库如果是本地需使用`file:///`协议。 限制（它指浅克隆版本库）： * 不能从它再克隆。 * 不能从它获取提交。 * 不能推送提交给它。 * 不要从它推送给其它版本库。 * 合并如无历史会有大量冲突。 `.git/shallow`列出了应该被视为根节点的提交ID。 ### 提交嫁接与替换 通过文件`.git/info/grafts`把完全不同的提交分支嫁接到一起。 格式：`<commit> <parent> [<parent>]*` 通过`.git/refs/replace/`下面的引用实现提交的替换。 `git replace` 管理提交替换。 ### 评注 位置`.git/ref/notes/`，评注记录在一个blob对象中，并以所评注的对象ID作为文件名保存在一个目录树中。包含所有评注及目录树的更改被提交到一个特殊的引用`.git/ref/notes/commits`之中。 `git notes` 评注操作。`list`列出。`show`显示。`add`添加。`append`追加。`copy`复制。`edit`编辑。`remove`删除。`merge`合并。 配置`notes.rewriteRef`, `notes.rewrite.amend`, `notes.rewrite.rebase`使提交评注随提交的修改而复制到新的提交之上。 ### 其它命令 命令参数中为了避免commit或引用和path冲突，中间要用`--`分隔。 `git show` 相当于`git cat-file -p`。 `git clean` `-fd`清除未加入版本库的文件和目录。`-nd`测试运行。 `git archive` 将版本库所有文件打包归档，如果要忽略某些文件可以设置export-ignore属性。`--prefix=`增加目录前缀。 `git check-ref-format` 检查引用的命名规范。 `git cherry` 查看哪些提交未被推送到上游跟踪分支中。 `git instaweb` 临时启动一个web访问实例。 `git request-pull` 生成等待更新(pull)的概要。 `git filter-branch` 针对不同的过滤器提供脚本命令从不同角度对版本库进行重构。 `git fast-import` 版本库导入。 `git help` 显示帮助。 ## 工具 图形化 * [gitk](http://git-scm.com/docs/gitk): 自带，基于Tcl/Tk * [gitg](https://wiki.gnome.org/Apps/Gitg): 基于GTK+ * [QGit](http://digilander.libero.it/mcostalba/): 基于Qt *
